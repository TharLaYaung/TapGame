# コードコメント作成ルール

今後の開発では、以下のルールに従ってコード内コメントを作成・修正してください。

## 基本方針
- コメントは「なぜその処理が必要か」を説明する
- コードを読めば分かる内容はコメントしない
- 実装内容の説明より、意図・理由・制約を書く
- コメントとコード内容がズレないようにする
- 冗長なコメントは禁止

## 禁止事項
- 「変数に代入する」など自明な説明
- 日本語と英語の混在ルール崩れ
- 古い仕様情報を残すこと
- 感情的・個人的なコメント
- TODOの放置

## 設計・保守ルール
- 関数コメントには「入力」「出力」「副作用」を記載
- 一時対応は期限・理由を明記
- ハックコードには背景を書く
- 外部仕様依存は参照元を記載
- セキュリティ関連処理は理由を明記

## コメント優先順位
以下を優先してコメントする：
1. バグ回避
2. 仕様制約
3. パフォーマンス理由
4. セキュリティ理由
5. 業務ルール

## 出力ルール
- コメントは簡潔に
- 1コメントは最大2行まで
- コードレビューで読まれる前提で書く
- 将来の保守担当者向けに書く
